## 🔴 CRITIQUES — à régler avant prod

### 1. Bypass admin sans contrôle serveur (sécurité)

**Fichier :** `src/lib/firestore-actions.ts:1177` + `src/components/tournee-scan-flow.tsx:147`

Le toggle "🔓 Mode admin (scanner dans le désordre)" que j'ai ajouté ce soir n'est rendu QUE pour les rôles admin/superadmin côté UI, mais le serveur accepte le flag `bypassOrderLock: true` tel quel sans vérifier le rôle de l'appelant. Un user staff non-admin avec console DevTools peut envoyer le flag manuellement et tout bypasser.

**Fix :** dans le case `markVeloPrepare/Charge/LivreScan/uploadMontagePhoto`, lookup `equipe/{userId}.role` et n'accepter le flag que si role ∈ {admin, superadmin}. +1 read par scan, raisonnable.

### 2. Hard-deletes qui violent la règle soft-cancel

**Fichiers :**
- `firestore-actions.ts:3979` `deleteClient` → `deleteDoc(doc(db, "clients", id))` direct
- `firestore-actions.ts:4229` `deleteVerification` → `deleteDoc(...)` direct
- `firestore-actions.ts:467-475` `setClientVelosTarget` → `batch.delete(v.ref)` sur les vélos en surplus

Memory `never_hard_delete_data.md` interdit explicitement ces suppressions.

**Fix :** poser `statut="annule"` ou `annule=true`, conserver l'enregistrement. La logique de filtrage côté UI ignore déjà les annulés dans la plupart des écrans.

### 3. gas-inbox sur clé Gemini free tier (à confirmer)

**Memory :** `crm_velos_cargo_gemini_billing.md` dit "gas-inbox/ (velos-cargo@) free tier au 2026-04-26". Statut actuel inconnu côté code.

**Risque :** si tu reçois >15 mails AXDIS/min, classifications bloquées 429.

**Fix :** activer billing sur la clé Gemini de velos-cargo@, ou basculer gas-inbox sur la clé yoann@ déjà billed. 5 min de config Apps Script Properties.

### 4. Cloud Function `gemini` HTTP publique

**Fichier :** `functions/src/index.ts:116` (`export const gemini = onRequest(...)`)

Endpoint HTTP sans auth. Tout internet peut consommer la clé Gemini → abus, 429, coûts. CORS appliqué mais insuffisant.

**Fix :** soit migrer en `onCall` (auth automatique), soit ajouter check `request.auth` après vérification d'un token. Le frontend appelle déjà `callGemini()` qui peut s'adapter au format `onCall`.

### 5. `syncDrive` et `markVeloMonte` non migrées Firestore

**Fichiers :**
- `clients/page.tsx:1484` appelle `gasPost("syncDrive", {})` (action GAS legacy)
- `montage/page.tsx:560` `_legacyMarkVeloMonteOnePhoto` (commenté en dead code mais rappelable)

Si GAS plante, ces actions plantent silencieusement. Le flow montage actuel (via `uploadMontagePhoto`) est OK, mais `syncDrive` reste utilisé côté admin.

**Fix :** migrer `syncDrive` en case Firestore + supprimer le dead code `_legacyMarkVeloMonteOnePhoto`.

### 6. Page `/bl` peut rester bloquée sur "Chargement…"

**Fichier :** `src/app/bl/page.tsx:44-51`

Si `gasGet("getBlForTournee")` plante silencieusement (timeout réseau, 4G zone blanche), `data` reste null et la page bloque indéfiniment. Bouton "Imprimer" jamais affiché.

**Fix :** ajouter un timeout (~10s) + state `hasError` + bouton "Réessayer".

### 7. Race condition latente sur le verrou LIFO inter-clients

**Fichier :** `firestore-actions.ts:1309` (`} catch { ... }`)

Le check d'ordre LIFO est dans un try/catch silencieux. Si Firestore plante en plein check (perte réseau), on **laisse passer** le scan. Côté défensif intentionnel mais il faudrait au moins logger.

**Fix :** ajouter `console.warn("[markVelo] LIFO check failed:", err)` dans le catch + remonter le warning au frontend pour info.
